206/// The shared state object for Future and Promise.
207///
208/// Nomenclature:
209///
210/// - "result": a `Try` object which, when set, contains a `T` or exception.
211/// - "move-out the result": used to mean the `Try` object and/or its contents
212/// are moved-out by a move-constructor or move-assignment. After the result
213/// is set, Core itself never modifies (including moving out) the result;
214/// however the docs refer to both since caller-code can move-out the result
215/// implicitly (see below for examples) whereas other types of modifications
216/// are more explicit in the caller code.
217/// - "callback": a function provided by the future which Core may invoke. The
218/// thread in which the callback is invoked depends on the executor; if there
219/// is no executor or an inline executor the thread-choice depends on timing.
220/// - "executor": an object which may in the future invoke a provided function
221/// (some executors may, as a matter of policy, simply destroy provided
222/// functions without executing them).
223/// - "consumer thread": the thread which currently owns the Future and which
224/// may provide the callback and/or the interrupt.
225/// - "producer thread": the thread which owns the Future and which may provide
226/// the result and which may provide the interrupt handler.
227/// - "interrupt": if provided, an object managed by (if non-empty)
228/// `exception_wrapper`.
229/// - "interrupt handler": if provided, a function-object passed to
230/// `promise.setInterruptHandler()`. Core invokes the interrupt handler with
231/// the interrupt when both are provided (and, best effort, if there is not
232/// yet any result).
233///
234/// Core holds three sets of data, each of which is concurrency-controlled:
235///
236/// - The primary producer-to-consumer info-flow: this info includes the result,
237/// callback, executor, and a priority for running the callback. Management of
238/// and concurrency control for this info is by an FSM based on `enum class
239/// State`. All state transitions are atomic; other producer-to-consumer data
240/// is sometimes modified within those transitions; see below for details.
241/// - The consumer-to-producer interrupt-request flow: this info includes an
242/// interrupt-handler and an interrupt. Concurrency of this info is controlled
243/// by a Spin Lock (`interruptLock_`).
244/// - Lifetime control info: this includes two reference counts, both which are
245/// internally synchronized (atomic).
246///
247/// The FSM to manage the primary producer-to-consumer info-flow has these
248/// allowed (atomic) transitions:
249///
250/// +----------------------------------------------------------------+
251/// | ---> OnlyResult ----- |
252/// | / \ |
253/// | (setResult()) (setCallback()) |
254/// | / \ |
255/// | Start ---------> ------> Done |
256/// | \ \ / |
257/// | \ (setCallback()) (setResult()) |
258/// | \ \ / |
259/// | \ ---> OnlyCallback --- |
260/// | \ or OnlyCallbackAllowInline |
261/// | \ \ |
262/// | (setProxy()) (setProxy()) |
263/// | \ \ |
264/// | \ ------> Empty |
265/// | \ / |
266/// | \ (setCallback()) |
267/// | \ / |
268/// | --------> Proxy ---------- |
269/// +----------------------------------------------------------------+
270///
271/// States and the corresponding producer-to-consumer data status & ownership:
272///
273/// - Start: has neither result nor callback. While in this state, the producer
274/// thread may set the result (`setResult()`) or the consumer thread may set
275/// the callback (`setCallback()`).
276/// - OnlyResult: producer thread has set the result and must never access it.
277/// The result is logically owned by, and possibly modified or moved-out by,
278/// the consumer thread. Callers of the future object can do arbitrary
279/// modifications, including moving-out, via continuations or via non-const
280/// and/or rvalue-qualified `future.result()`, `future.value()`, etc.
281/// Future/SemiFuture proper also move-out the result in some cases, e.g.,
282/// in `wait()`, `get()`, when passing through values or results from core to
283/// core, as `then-value` and `then-error`, etc.
284/// - OnlyCallback: consumer thread has set a callback/continuation. From this
285/// point forward only the producer thread can safely access that callback
286/// (see `setResult()` and `doCallback()` where the producer thread can both
287/// read and modify the callback).
288/// - OnlyCallbackAllowInline: as for OnlyCallback but the core is allowed to
289/// run the callback inline with the setResult call, and therefore in the
290/// execution context and on the executor that executed the callback on the
291/// previous core, rather than adding the callback to the current Core's
292/// executor. This will only happen if the executor on which the previous
293/// callback is executing, and on which it is calling setResult, is the same
294/// as the executor the current core would add the callback to.
295/// - Proxy: producer thread has set a proxy core which the callback should be
296/// proxied to.
297/// - Done: callback can be safely accessed only within `doCallback()`, which

329// exception_ptr_get_object
330//
331// Returns the address of the stored exception as if it were upcast to the
332// given type, if it could be upcast to that type.
333template <typename T>
334T* exception_ptr_get_object(std::exception_ptr const& ptr) noexcept {
335 static_assert(!std::is_reference<T>::value, "is a reference");
336 auto target = type_info_of<T>();
337 auto object = !target ? nullptr : exception_ptr_get_object(ptr, target);
338 return static_cast<T*>(object);
