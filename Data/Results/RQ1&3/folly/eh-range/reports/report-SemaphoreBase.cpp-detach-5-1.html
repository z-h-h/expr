76/// Promises and futures provide a potentially nonblocking mechanism
77/// to execute a producer/consumer operation concurrently, with
78/// threading/pools controlled via an executor. There are multiple potential
79/// patterns for using promises and futures including some that block the
80/// caller, though that is discouraged; it should be used only when necessary.
81///
82/// One typical pattern uses a series of calls to set up a small, limited
83/// program that...
84///
85/// - ...performs the desired operations (based on a lambda)...
86/// - ...on an asynchronously provided input (an exception or a value)...
87/// - ...lazily, when that input is ready (without blocking the caller)...
88/// - ...using appropriate execution resources (determined by the executor)...
89/// - ...then after constructing the 'program,' launches the asynchronous
90/// producer.
91///
92/// That usage pattern looks roughly like this:
93///
94/// auto [p, f] = makePromiseContract(executor);
95/// g = std::move(f).then([](MyValue&& x) {
96/// ...executor runs this code if/when a MyValue is ready...
97/// });
98/// ...launch the async producer that eventually calls p.setResult()...
99///
100/// This is just one of many potential usage patterns. It has the desired
101/// property of being nonblocking to the caller. Of course the `.then()`
102/// code is deferred until the produced value (or exception) is ready,
103/// but no code actually blocks pending completion of other operations.
104///
105/// The promise/future mechanism is limited to a single object of some arbitrary
106/// type. It also supports a (logically) void result, i.e., in cases where the
107/// continuation/consumer (the `.then()` code if using the above pattern) is not
108/// expecting a value because the 'producer' is running for its side-effects.
109///
110/// The primary data movement is from producer to consumer, however Promise and
111/// Future also provide a mechanism where the consumer can send an interruption
112/// message to the producer. The meaning and response to that interruption
113/// message is controlled by the promise; see `Promise::setInterruptHandler()`.
114///
115/// Neither Promise nor Future is thread-safe. All internal interactions
116/// between a promise and its associated future are thread-safe, provided that
117/// callers otherwise honor the promise's contract and the future's contract.
118///
119/// Logically there are up to three threads (though in practice there are often
120/// fewer - one thread might take on more than one role):
121///
122/// - Set-up thread: thread used to construct the Promise, and often also to
123/// set up the SemiFuture/Future.
124/// - Producer thread: thread that produces the result.
125/// - Consumer thread: thread in which the continuation is invoked (a
126/// continuation is a callback provided to `.then` or to a variant).
127///
128/// For description purposes, the term 'shared state' is used to describe the
129/// logical state shared by the promise and the future. This 'third object'
130/// represents things like whether the result has been fulfilled, the value or
131/// exception in that result, and the data needed to handle interruption
132/// requests.
133///
134/// A promise can be in various logical states:
135///
136/// - valid vs. invalid (has vs. does not have a shared state, respectfully).
137/// - fulfilled vs. unfulfilled (an invalid promise is always fulfilled; a valid
138/// promise is fulfilled if the shared-state has a result).
139///
140/// A promise `p` may optionally have an associated future. This future, if it
141/// exists, may be either a SemiFuture or a Future, and is defined as the
142/// future (if any) that holds the same shared state as promise `p`.
143/// The associated future is initially the future returned from
144/// `p.getFuture()` or `p.getSemiFuture()`, but various operations
145/// may transfer the shared state from one future to another.

206/// The shared state object for Future and Promise.
207///
208/// Nomenclature:
209///
210/// - "result": a `Try` object which, when set, contains a `T` or exception.
211/// - "move-out the result": used to mean the `Try` object and/or its contents
212/// are moved-out by a move-constructor or move-assignment. After the result
213/// is set, Core itself never modifies (including moving out) the result;
214/// however the docs refer to both since caller-code can move-out the result
215/// implicitly (see below for examples) whereas other types of modifications
216/// are more explicit in the caller code.
217/// - "callback": a function provided by the future which Core may invoke. The
218/// thread in which the callback is invoked depends on the executor; if there
219/// is no executor or an inline executor the thread-choice depends on timing.
220/// - "executor": an object which may in the future invoke a provided function
221/// (some executors may, as a matter of policy, simply destroy provided
222/// functions without executing them).
223/// - "consumer thread": the thread which currently owns the Future and which
224/// may provide the callback and/or the interrupt.
225/// - "producer thread": the thread which owns the Future and which may provide
226/// the result and which may provide the interrupt handler.
227/// - "interrupt": if provided, an object managed by (if non-empty)
228/// `exception_wrapper`.
229/// - "interrupt handler": if provided, a function-object passed to
230/// `promise.setInterruptHandler()`. Core invokes the interrupt handler with
231/// the interrupt when both are provided (and, best effort, if there is not
232/// yet any result).
233///
234/// Core holds three sets of data, each of which is concurrency-controlled:
235///
236/// - The primary producer-to-consumer info-flow: this info includes the result,
237/// callback, executor, and a priority for running the callback. Management of
238/// and concurrency control for this info is by an FSM based on `enum class
239/// State`. All state transitions are atomic; other producer-to-consumer data
240/// is sometimes modified within those transitions; see below for details.
241/// - The consumer-to-producer interrupt-request flow: this info includes an
242/// interrupt-handler and an interrupt. Concurrency of this info is controlled
243/// by a Spin Lock (`interruptLock_`).
244/// - Lifetime control info: this includes two reference counts, both which are
245/// internally synchronized (atomic).
246///
247/// The FSM to manage the primary producer-to-consumer info-flow has these
248/// allowed (atomic) transitions:
249///
250/// +----------------------------------------------------------------+
251/// | ---> OnlyResult ----- |
252/// | / \ |
253/// | (setResult()) (setCallback()) |
254/// | / \ |
255/// | Start ---------> ------> Done |
256/// | \ \ / |
257/// | \ (setCallback()) (setResult()) |
258/// | \ \ / |
259/// | \ ---> OnlyCallback --- |
260/// | \ or OnlyCallbackAllowInline |
261/// | \ \ |
262/// | (setProxy()) (setProxy()) |
263/// | \ \ |
264/// | \ ------> Empty |
265/// | \ / |
266/// | \ (setCallback()) |
267/// | \ / |
268/// | --------> Proxy ---------- |
269/// +----------------------------------------------------------------+
270///
271/// States and the corresponding producer-to-consumer data status & ownership:
272///
273/// - Start: has neither result nor callback. While in this state, the producer
274/// thread may set the result (`setResult()`) or the consumer thread may set
275/// the callback (`setCallback()`).
276/// - OnlyResult: producer thread has set the result and must never access it.
277/// The result is logically owned by, and possibly modified or moved-out by,
278/// the consumer thread. Callers of the future object can do arbitrary
279/// modifications, including moving-out, via continuations or via non-const
280/// and/or rvalue-qualified `future.result()`, `future.value()`, etc.
281/// Future/SemiFuture proper also move-out the result in some cases, e.g.,
282/// in `wait()`, `get()`, when passing through values or results from core to
283/// core, as `then-value` and `then-error`, etc.
284/// - OnlyCallback: consumer thread has set a callback/continuation. From this
285/// point forward only the producer thread can safely access that callback
286/// (see `setResult()` and `doCallback()` where the producer thread can both
287/// read and modify the callback).
288/// - OnlyCallbackAllowInline: as for OnlyCallback but the core is allowed to
289/// run the callback inline with the setResult call, and therefore in the
290/// execution context and on the executor that executed the callback on the
291/// previous core, rather than adding the callback to the current Core's
292/// executor. This will only happen if the executor on which the previous
293/// callback is executing, and on which it is calling setResult, is the same
294/// as the executor the current core would add the callback to.
295/// - Proxy: producer thread has set a proxy core which the callback should be
296/// proxied to.
297/// - Done: callback can be safely accessed only within `doCallback()`, which
298/// gets called on exactly one thread exactly once just after the transition
299/// to Done. The future object will have determined whether that callback
300/// has/will move-out the result, but either way the result remains logically
301/// owned exclusively by the consumer thread (the code of Future/SemiFuture,
302/// of the continuation, and/or of callers of `future.result()`, etc.).
303/// - Empty: the core successfully proxied the callback and is now empty.
304///
305/// Start state:
306///
307/// - Start: e.g., `Core<X>::make()`.
308/// - (See also `Core<X>::make(x)` which logically transitions Start =>
309/// OnlyResult within the underlying constructor.)
310///
311/// Terminal states:
312///
313/// - OnlyResult: a terminal state when a callback is never attached, and also
314/// sometimes when a callback is provided, e.g., sometimes when
315/// `future.wait()` and/or `future.get()` are used.
316/// - Done: a terminal state when `future.then()` is used, and sometimes also
317/// when `future.wait()` and/or `future.get()` are used.
318/// - Proxy: a terminal state if proxy core was set, but callback was never set.
319/// - Empty: a terminal state when proxying a callback was successful.
320///
321/// Notes and caveats:
322///
323/// - Unfortunately, there are things that users can do to break concurrency and
324/// we can't detect that. However users should be ok if they follow move
